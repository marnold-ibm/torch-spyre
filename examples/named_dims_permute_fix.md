# Fix: named dim propagation through permuted intermediate buffers

## Background

`compute_input_named_dims` maps loop variables to named dim names for each
buffer read by an op.  The core step is calling `compute_coordinates(size,
stride, ranges, dep.index)` to invert `dep.index` into one coordinate
expression per dim.  Each coordinate should contain exactly one free symbol,
which identifies the loop variable for that dim.

`dep.index` is always generated by Inductor against the buffer's **physical**
strides, so `compute_coordinates` must receive those same strides to invert it
correctly.

## The bug

The old code passed **canonical** strides to `compute_coordinates` — strides
derived from the declared named-dim sizes, assuming contiguous storage in
declaration order.  For a graph input annotated in physical storage order these
happen to match, so the inversion was accidentally correct.

For a **permuted intermediate buffer** the physical strides differ from
canonical.  Example: `queries[B,Lq,H,D].permute(0,2,1,3) * scale` produces
`buf1` with named dims `['B','H','Lq','D']` but physical strides that reflect
the permuted layout:

```
buf1 physical strides: [1048576, 128, 4096, 1]   ← H at stride 128, Lq at 4096
canonical strides:     [1048576, 32768, 128, 1]  ← would be H at 32768, Lq at 128

dep.index = 1048576*d0 + 128*d1 + 4096*d2 + d3

compute_coordinates with canonical strides:
  H coordinate = d1 + 32*Mod(d2, 8)   ← two free symbols {d1, d2}

_lone_sym() picks one from frozenset{d1, d2} non-deterministically
→ wrong named dim assignment, different result each run
```

## The fix

Always use physical strides via `host_coordinates`.  This correctly inverts
`dep.index` into one single-symbol coordinate per physical dim.

### View/reshape buffers

When a buffer was created by collapsing multiple named dims into one physical
dim (e.g. `w[A, B*D*E]` annotated `['A','B','D','E']`), one physical dim maps
to several named dims.  Its coordinate contains multiple free symbols — one per
collapsed named dim.

`_decompose_coord` detects this and sub-decomposes using the canonical strides
of just those named dims.  This works because a proper annotation guarantees
the named dims tile that physical dim contiguously, so the sub-strides match
exactly.

Example: `buf0` has shape `[1, 2, 384]` because `w+x` over `w[1,A,B*D*E]`
produces a 3-dim output, but it carries named dims `['A','B','D','E']`.  When
op1 reads it with `dep.index = 384*d0 + 128*d2 + 64*d3 + d4`:

```
host_coordinates([1,2,384], strides=[768,384,1], dep.index)
  → [0,  d0,  128*d2 + 64*d3 + d4]
       ↑   ↑   ↑
       |   |   3 symbols → sub-decompose with named dims ['B','D','E']
       |   single symbol → d0 maps to 'A'
       constant (trivial batch dim, no named dim)

sub-decompose(128*d2 + 64*d3 + d4, ['B','D','E'], canonical strides [128,64,1])
  → d2 → 'B',  d3 → 'D',  d4 → 'E'

final mapping:  d0→A   d2→B   d3→D   d4→E
```

### Graph inputs

`_get_buffer` returns `None` for graph inputs (they live in `graph_inputs`, not
`graph.buffers`).  Graph inputs to `torch.compile` are always contiguous, so
canonical strides equal physical strides and the fallback is correct.

## Algorithm

```
phys_coords = host_coordinates(layout, dep)   # one coord per physical dim

for each (phys_coord, named_dims_for_this_physical_dim):
    if phys_coord has 1 free symbol:
        map that symbol → named dim            # normal case
    else:
        sub-decompose using canonical strides of those named dims
        map each resulting symbol → its named dim   # view/reshape case
```

The multi-symbol coordinate is the natural signal for the view/reshape case —
no branching on dimension counts needed.
